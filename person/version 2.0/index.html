Search Bar Usage

Description

The search bar provides the ability to quickly navigate to an entity in the listing.

Usage

In the "Search for entities" bar, begin typing the name of an entity of interest.

A dropdown will display suggestions based upon the text in the search bar.

To select the entity from the dropdown, use the down arrow key to navigate to the particular entity, then press tab for the selected entity to fill in the text field.

Finally, press enter and the entity list will jump to the selected entity. To view that entitiy's documentation, press enter again or click on the entity in the list

Note: Queries with the search bar are case insensitive.

Filter Bar Usage

Description

The filter bar provides the ability to filter the list of entities by attribute.

Usage

The mechanics of using the filter bar are the same as the search bar mechanics described above.

The filter bar supports complex filters using the keywords "and", "or" and "not" and parentheses "(" and ")".

Try the example filter below: 

 ((startOn or endOn) and metadata)

Note: Queries with the search bar are case sensitive.

The REST APIs provide programmatic access to read and write Ellucian Higher Education Data Model data. The Ellucian Higher Education Data Model (HEDM) is a single model of all the things and concepts in the entire domain of higher education administration. Each thing/concept is treated as a separate entity in the model, and the same name is used by both business and technical audiences. The model is expressed in the form of JSON Schema documents.

Use of HTTP

Status: Under Review

Since: version 0.2.0 Note: Specific sections may explicitly identify a later version. The specified version above represents the 'default' HEDM version for which this standard applies.

HEDM HTTP APIs must comply with the standards specified herein.

The API Strategy may be referenced for additional background information. This document supercedes the API Strategy, as that document is not HEDM specific.

This document establishes HEDM standards for using HTTP.

HTTP Methods

APIs must expose a standard REST interface using the HTTP methods (aka verbs) shown below:

HTTP Method Purpose
GET Read a resource or list of resources
POST Create a new resource (when the key is not known a-priori)
PUT Update an existing resource or create one if the key is pre-defined
PATCH Partially update an existing resource
DELETE Remove a resource

HTTP PATCH support was not required in earlier HEDM versions. Consequently, this support must be added for existing APIs to be compliant with this specification.

HEDM API support for HTTP PATCH must also support JSON Patch. JSON Patch provides a very explicit mechanism to describe changes to a resource. The proper custom media type for HTTP PATCH requests using JSON Patch is 'application/json-patch+json'. This standard custom media type must be set as the Content-Type header irrespective of the resource being patched.

Safety and idempotency:

  • HTTP 'GET' must be 'safe', meaning such requests must not have side effects and must not modify state on the server.
  • HTTP 'GET', 'PUT', 'PATCH' and 'DELETE' must be 'idempotent', meaning subsequent invocations of the same request must not have any adverse consequences (other than processing overhead in order to provide a response to the client).

When using PUT to 'update' a resource, PUT support must allow for partial updates. That is, while PUT semantics are to 'replace' a resource, HEDM APIs must support requests containing only required properties (i.e., they must not remove optional properties from an existing resource if not included in a PUT request).

Two additional HTTP methods are encouraged but not required (as some frameworks do not provide support for them). These are:

HTTP Method Purpose
HEAD Read metadata about a resource, but not the resource itself
OPTIONS Discover the supported methods, as returned in an 'Allow' header

Support for OPTIONS is particularly important when using CORS, to support pre-flight requests issued by modern browsers.

HTTP Status Codes

HEDM HTTP/S APIs must use standard HTTP status codes as specified in the table below:

Status Code Description
200 The request was successful.
201 A resource was successfully created.
202 Request has been received. This is only used for asynchronous processing.
204 Request was successful and no content is returned. This is a common response to a DELETE request, and is permitted for POST if there is no value in returning the newly created resource.
304 The GET request was conditional and contained an 'If-None-Match' header that was used to determine the current representation has not changed
400 The request cannot be understood. This should be used if there are syntax errors and validation errors, ('422' is also acceptable for validation errors.)
401 Authentication has failed, or the authenticated client is not authorized.
404 The requested resource could not be found.
405 The method is not supported by the resource.
406 Not Acceptable. Return this code when the Accept header (or a custom header as discussed in this strategy) requests a response in an unsupported format.
409 There is a conflict. Return this if the request reflects stale data (that violates an optimistic lock).
412 A precondition failed. Used for conditional requests (e.g., when an 'If-Match' header does not match a resource's Etag value).
415 Unsupported Media Type. Return when a client request contains content in an unsupported format.
417 Expectation Failed. Return this if appropriate when supporting 'look-before-you-leap' requests
429 The rate limit is exceeded
500 An undisclosed server error occurred. This is a generic 'fail whale' response.

Note: The above table represents a minimum list for API developers, your middleware will certainly use additional codes, particularly those related to redirects.

Validation Errors

Responses must report validation errors using HTTP Status Code '400', and must return a body containing details of the error.

Following is an example error:

  { "errors" : [
							{
							  "type": "validation",
							  "resource": {

Conditional Requests and Caching

Since: 0.5.0     Note: HEDM APIs may add support for ETag and conditional requests as described in this section prior to this identified release, however they must not require use of conditional requests until this release. This is a breaking change from the previous release.

HEDM HTTP APIs must include an ETag header within a response to each GET request.

A HEDM API must support conditional GET requests that use the 'If-None-Match' header containing an ETag value returned from a previous GET request. If the service finds the resource still has a representation that has this ETag, the service must return an HTTP status code of 304 instead of returning the resource.

Similarly, a HEDM API must support conditional PUT or PATCH requests. If the 'If-Match' header contains an ETag that is different than an ETag calculated for a representation based on the current state of the resource, an HTTP status code of 412 should be returned.

If an API client retrieves a list of some resource (e.g., a 'page'), that list must contain an ETag. That ETag, however, is only useful to support subsequent conditional GET requests of that same 'page'. That is, an ETag for a list needs to account for all elements within that list, and is not the same ETag associated with any particular element contained in that list. In fact, an ETag for a list can be a hash of all the ETags for the elements in the list.

Consequently, a client will not be able to issue a PUT or PATCH request for a resource without first retrieving the affected resource explicitly (i.e., individually) so that it can attain the needed ETag corresponding to that specific resource.

If an 'If-Match' header is not provided within a PUT or PATCH request, a '412' must be returned.

Optimistic Locking

Resources are often protected against stale updates by incorporating an optimistic locking strategy. This is not surfaced using HEDM APIs, as an optimistic lock token (aka revision or version) is not included within HEDM models.

The use of ETags and conditional requests (as discussed above) will be used to prevent 'stale' updates. When implemented, an ETag may be constructed by including the optimistic lock tokens when available within the underlying datastore.

URI conventions

  • Use transparent (i.e., readable) URIs unless specific requirements preclude
  • Use dashes (aka hyphens) in lieu of underscores
  • Use camelCase when referencing property names
  • Use pluralized resource names for URLs, regardless of whether a collection or a single instance is being addressed. For example, a list of colleges might be retrieved using 'GET /colleges' while a specific college instance might be retrieved using 'GET /colleges/{id}'.

Versioning

HEDM APIs must use custom media types within the 'Accept' and 'Content-Type' HTTP Headers to identify the (major) version of HEDM being requested or used.That is, a HEDM API is deployed to support one or more major versions of HEDM.

HEDM JSON Schema Catalog releases are versioned using Semantic Versioning (version 2), and an API may support one or more major releases of the catalog. Only 'major' versions are identified within a custom media type (as 'minor' and 'patch' changes are by definition backward compatible, a HEDM API implementation is free to adopt the last backward compatible version to satisfy a request identifying a major version).

As an example, let us assume HEDM JSON Schema Catalog 3.4.17 is released and is adopted by a HEDM API that was previously supporting catalog version 3.3.19. HTTP requests to that API including a custom media type identifying 'v3' will now use this 'new' catalog release.

A HEDM API may support multiple HEDM versions, however this support is limited to a single version per major release. That is, a HEDM API that supports 'v3' may also support 'v2' (which would use the latest HEDM JSON Catalog 2.x.x release available).

When using custom media types, they do not need to be registered nor do they need to correspond to a schema document (although they often may).

Media Types

An HTTP request should include HTTP 'Content-Type' and 'Accept' headers that identify the media type used in the request and the media type desired in the response, respectively.

A custom media type may be used to identify a specific representation of a resource:

    application/vnd.hedtech.supercourses.v4+json

To identify a default representation (one that corresponds to the HEDM JSON Schema for that resource) and indicate a particular (major) version (e.g., 3.x.x), a more general custom media type may be employed:

    application/vnd.hedtech.hedm.v3+json

This more general 'hedm' custom media type is preferred when requesting default representations. There is no benefit to identifying a resource within the custom media name if the default representation is requested, as the resource name is already provided within the request's URI.

Lastly, the client may use a standard media type if identifying 'the latest' supported version using the default representation:

    application/json

When a request identifies a custom media type that is not supported, a 415 (unsupported media type) error should be returned.

Dates and Times

HEDM model properties using a JSON Schema 'format' of ”date-time” must contain values that adhere to RFC3339 section 5.6. (Note RFC3339 is a profile of ISO .)

This same RFC applies to specifying dates and times on the URI when used as query paramters.

HEDM Bulk Data

Status: Accepted

Since: version 0.2.0

Bulk processing provides a way to apply one API call to multiple instances of the same resource (e.g., HEDM model), all in a single payload. The request payload is a JSON array of instances of the same resource (e.g., persons). The response payload is a JSON array of results, in exactly the same order as requested. That is, if a bulk request consists of 10 'persons', then the response will contain a JSON array of ten results, in the same order. There is no explicit index; it is the responsibility of the service to ensure the results array is in the same order.

Each result within the response's JSON array will be either a full representation of the instance (if that instance was created or updated successfully) or an error document that describes why the operation on that instance was not successful. All bulk requests are "greedy", as many instances in the request payload that can be accepted will be.

Bulk APIs may normally synchronous, although an asynchronous API may be implemented in compliance with the Batch API Standard currently under development. This standard focuses on synchronous bulk APIs.

Example:

Request
{
						{person instance 1},
						{person instance 2},
						{person instance 3}
					}
Response:
{
						{person instance 1 full representation},  // accepted
						{error for instance 2},  // not accepted
						{person instance 3 full representation} // accepted
					}

In this example, instances 1 and 3 of the bulk request were complete successfully, but instance 2 was not.

API Filtering

Status: Proposed

Since: version 0.2.0

HEDM APIs may need to support filtering through the use of 'ad-hoc' and or 'named' queries. When such filtering is needed for an API, it must be implemented in accordance to this specification. The determination of filtering requirements, and whether ad-hoc or named queries should be employed, must be determined by the HEDM working group.

Filtering capability must be specified in documentation for each HEDM model, and all solutions that may be configured as an authoritative source for that model must support the specified filtering.

Ad-Hoc Query Support

Ad-hoc queries must use the query syntax described herein, which is a subset of the MongoDB query syntax. The rest of this page describes the specific parts that will be supported by Ellucian. Additional part of the MongoDB syntax can be adopted if deemed necessary. Using this syntax, queries are defined as a JSON document. Queries are submitted by passing the query document as a JSON-formatted string using the a query parameter called "filter". For example:

http://localhost/fsu/api/courses?filter={"title" : "my course title"}

In the above example, the query document is shown as plain text for the sake of clarity. However, in reality this string should be URL encoded. For example:

http://localhost/fsu/api/courses?filter=%7B%22title%22%3A%22my%20course%20title%22%7D

The following sections provide an overview of the query syntax.

Equality

The simplest query is a test for equality.

{"title" : "MATH101"}

The above example says give me all the courses where the course title equals MATH101. You can create an AND query by specifying more than one property in the query document. The following example says return all instances where description equals Beginning mathematics and number equals 101.

{"description" : "Beginning mathematics", "number" : "101"}
Basic Operators

The following basic operators are supported.

Operator Definition
$gt Greater than
$gte Greater than or equal
$lt Less than
$lte Less than or equal
$ne Not equal

Below is an example of how to use the above operators. This example say give me all the instances where count is greater than 10. 

{"count" : {"$gt" : 10}}
OR Queries

An OR query can be performed using the $or operator. The example below will return all instances where property1 equals true or property equals false.

{"$or" : [{"property1" : true}, {"property2" : false}]}

The $or operator can also be used in conjunction with the other operators. For example:

{"$or" : [{"qty" : {"$gt : 10}}, {"qty" : {"$lt" : 20}}]}
Querying Arrays

Given the following document

{"id" : "53b2f1d0641a0a0f000c2f13", "colors" : ["red", "blue", "green"]}

To exactly match the array, both values and order, your query document would look like the following.

{"colors" : ["red", "blue", "green"]}

To match one or more elements of the array, without regard for position, you use the $all operator.

{“colors” : {“$all” : [“red, “green’]}}
Querying Embedded Documents

Given the following document.

{"id" : "53b2f1d0641a0a0f000c2f13", "item" : {"name" : "item1", "code" : "001"}}

To exactly match the entire embedded document, your query document would look like the following.

{"item" : {"name" : "item1", "code" : "001"}}

To match a single element of an embedded document use dot notation. For example:

{"item.code" : "001"}

Named Query Support

A named query is a pre-defined query that must be described within the API documentation. Named queries may be used to perform very complex queries without requiring the API user to construct a complicated ad-hoc query.

A named query is specified as a query parameter and may require arguments which must be expressed using JSON (where the arguments are provided as name-value pairs akin to the ad-hoc query syntax used for filtering by 'equality', as described above).

?highestPerformingStudents={"grade":"A","honors":"true"}

Query Specification

HEDM documentation pertaining to each HEDM model must specify the filtering requirements that must be supported by any authoritative source for that model. This model-specific documentation must specify which model properties may be included within an ad-hoc query, and specify the named queries (and arguments) that must be supported.

That is, while the implementation of a query may vary greatly by technology, the semantics and interface of that query must be consistent regardless of what solution is configured as the authoritative source for that model.

Note it is necessary to limit the properties that may be included within an ad-hoc query to ensure acceptable performance (e.g., indexing may be necessary within a datastore).

The HEDM working group must be used to coordinate the specification and agreement of specific filtering requirements for each HEDM model.

HEDM Pagination

Status: Accepted

Since: version 0.2.0

HEDM APIs supporting retrieval of multiple resources (e.g., 'GET /things') must employ 'offset-based' pagination as described below.

HTTP Request Query Parameters

Two URL query parameters must be accepted when handling a GET request for multiple resources. These query parameters allow a client to request subsets (aka pages or blocks) of a dataset at a time.

These two query parameters are:

Query Param description
offset This represents the offset from the first item in the dataset.
limit This represents the maximum number of items that may be included in the response.

It is imperative each HEDM API restrict the limit used to an appropriate maximum value. Specifically, each HEDM API must override the client-specified limit if that requested limit exceeds some threshold. The specific threshold is not specified, and will likely vary by resource.

For example, if the HEDM API has established an internal threshold of '100' for '/things', a client request that contains a 'limit' query parameter of '300' will not result in an error but the response will contain only '100' things.

HTTP Response Headers

While the two query parameters above are used to communicate pagination within a Request, a HEDM API is required to communicate pagination within the response. This is accomplished using two HTTP Headers, one standard and one custom, as described below.

X-Total-Count

This custom Header represents the total number of items in the dataset, not just the limited number returned in the response. For example, if a backing datastore contained a total of 43,000 students, the X-Total-Count when retrieving students (e.g., 100 at a time) would be '43000'.

This useful header may be omitted in cases where the total count is difficult to assertain. However, it is expected the great majority of HEDM APIs will be able to determine and return the total count.

Link

A standard Link Header must be used to communicate pagination within a response to a GET request for multiple resource instances that does not include the full dataset. (If it does include the full dataset (i.e., there is only one page) the Link Header may optionally still be included.)

This standard identifies four 'link relation types' ('first', 'prev', 'next', and 'last') that pertain to pagination. These are shown in the example Link Header below: 

Link: <https://{host:port}/courses?offset=500&limit=100>; rel="next",
					  <https://{host:port}/courses?offset=2700&limit=100>; rel="last",
					  <https://{host:port}/courses?offset=0&limit=100>; rel="first", 
					  <https://{host:port}/courses?offset=300&limit=100>; rel="prev"


HEDM APIs must support use of a concatenated Link header as defined at http://www.w3.org/Protocols/9707-link-header.html, where multiple lines need to be prefixed with a whitespace character. This is shown above.

Note each response may not contain all four of these link relations (aka relation types). For example, an initial request for the first 100 items will not be able to include a 'prev' link relation (as there is no previous batch of items at the beginning of the dataset). Similarly, the last batch of items would not be able to include the 'next' link relation.

Also note multiple link relations may be combined for one link. For instance, when the 'next' batch of items is also the 'last' batch, these two link relations may be expressed together as shown below:

<https://{host:port}/courses?offset=500&limit=100>; rel="next last"

If we're on the only 'page' (the entire dataset is returned, and contains 83 elements), the Link Header and X-Total-Count headers would look like:

Link: <https://{host:port}/fewThings?offset=0&limit=100>; rel="first last"
					X-Total-Count: 83


There are existing libraries to facilitate parsing of the Link Header available for most platforms.

Lastly, similar to the X-Total-Count custom header discussed above, if the total count cannot be assertained it will not be possible to include the 'last' link relation. This is acceptable (albeit undesirable), altough it is expected the great majority of HEDM APIs will be able to include the 'last' link relation.

Related Standards

Please see HEDM standards for 'filtering' and 'sorting'. Filtering, sorting and paging are often used together.

HEDM Error Message Schema

This describes the HEDM schema that enables conveying information about an error caused by, or related to, the request payload, from the recipient system to the requestor one.

Assumptions and Constraints

  • Systems communicate in the synchronous mode.
  • An HTTP error code is returned in the message envelope rather than in the message payload.
  • This schema is to only support business logic related error, when the data from the request is consumed and processed.
  • Ellucian will maintain one, master, list of all error messages that all publishers will agree to use. Publishers may request additional errors to be included in the list.
  • The master list of error messages will be documented as data and checked into the repository under its own directory. Publishers will provide meaningful explanations to each error for the entities, which they are assigned to be the authoritative source for (or the custodian of).
  • Individual schemas may include additional information about the interpretation of particular errors in their supplemental documentation.

Error Code Format

Error codes should be unique per resource and follow the dot notation pattern, e.g., "Room.Unavailable". There is a folder 'data/errors' where JSON files are located, which contain the list of all supported error codes by HEDM entity. Here are a few examples of possible entries:

    [
						  { 
							"code": "Instructor.Unavailable",
							"title": "Instructor Unavailable",
							"description": "The instructor selected for this event is not available within the chosen timeframe"
						  },
						  {
							"code": "Instructor.InvalidRole",
							"title": "Invalid Role for Instructor",
							"description": "The instructor selected for this event does not have a proper role to teach the course"
						  }
						]

Error Message Model

The error message model is a vehicle that delivers information about the exception that occurred in the recipient system to the requestor.

Error Message elements Required (Y/N)
An error code Y
A error human-readable error description N
The URL of the resource request that resulted in the error condition being triggered N*
A collection of JSON pointers to properties in the entities where the error occurred N**

* Required if the error is traceable to a specific resource; ** Required if the error is traceable to specific entities and their properties
